      マイグレーションツールの答えが出た（2026-03現在）

      claude で遊んでた時に DB Migration Tool を調査させたら sqldef を提案してきた。
      この手の汎用マイグレーションツールは、sql-migrate までしか知らなかったので調べてみた。

      従来型のマイグレーション

      RDBMS のマイグレーションツールあるあるで、差分をパッチで当てていくものが多い。この形の何が気に食わないのかと言うと、完全なDDLが管理されない。最新のDB定義はDB上にしかなく、ソース管理には最初に作ったものと差分が蓄積していく。
      Drizzle なんかがそう。generate でマイグレーションファイルが生成されるが、これは最新スナップショットとの差分であり、全量のDDLにはならない。

      また、DB上にどこまで適用されているかを保持する必要があり、Drizzle の場合 __drizzle_migrations のようなテーブルが作成される。sql-migrate も同様。

      migrations/
├── 0000_create_users.sql          -- CREATE TABLE users ...
├── 0001_add_email_to_users.sql    -- ALTER TABLE users ADD COLUMN email ...
├── 0002_create_posts.sql          -- CREATE TABLE posts ...
├── 0003_add_index_to_posts.sql    -- CREATE INDEX idx_posts_user_id ...
├── 0004_alter_users_add_role.sql  -- ALTER TABLE users ADD COLUMN role ...
└── ...                            -- 増え続ける

      sqldef は？

      sqldef は逆に、最新のDDLしか管理しない。DBへの反映時にDDLと現状のDBとの差を適用する形となっている。そのためマイグレーションファイルが増えない。
      これが DDL ベースの自動生成と相性が良く、go の sqlc-gen-crud のような定義から基本の CRUD を自動生成する ORM と組み合わせると捗る。
      DBの状態問わずDDLの状態と合わせにいくので、down migration もソースを revert するだけ。

      その時点のDBとの差分で反映するためDB上の適用状態を考慮する必要がない。

      db/
├── schema.sql        -- 全テーブルの CREATE TABLE（これが唯一の真実）
└── query/
    ├── users.sql     -- sqlc 用クエリ定義
    └── posts.sql

gen/
├── models.go         -- sqlc が schema.sql から自動生成
├── users.crud.go     -- sqlc-gen-crud が自動生成（CRUD）
├── posts.crud.go
├── users.sql.go      -- sqlc が query/*.sql から自動生成
└── posts.sql.go

      DDLのマスタが管理できて、マイグレーション都合の考慮が減るのが良き。

      その他

      そのまま使うとカラム名変更は drop column + add column になるが、これはコメントで補足することで alter 扱いになる。 Renaming Tables, Columns, and Indexes

      ただし、これ使うと適用状態の管理がちょっと煩雑化しそう。複数回の名称変更は避けた方が良さそう。
      そもそも SOLID の考え方上、変更加えるくらいなら、やむを得ない場合を除いてカラム追加して改修し切ってから古いカラム消す方が良さそう。
